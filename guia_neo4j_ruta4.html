Pregunta central: ¿Qué servicios de la infraestructura universitaria de IA se ven afectados cuando falla un servidor o un servicio clave, y a qué estudiantes impacta directamente?

Esta pregunta exige caminar relaciones a profundidad variable — exactamente el patrón que justifica un grafo.

Capítulo 1: Justificación de la Ruta

1.1 Problema y usuario

La infraestructura de IA de una universidad tiene servidores GPU que alojan servicios (JupyterHub, Ollama, VPN, vLLM…). Los estudiantes trabajan en proyectos que usan esos servicios. Los técnicos atienden tickets de incidencia.

El flujo de soporte necesita responder preguntas como:

  • Si el servidor H200 cae, ¿qué servicios dejan de funcionar?
  • ¿Qué estudiantes quedan bloqueados y cuáles son sus tickets activos?
  • ¿A través de qué cadena de dependencia se propagó la falla?

1.2 ¿Por qué Ruta 4 — Grafo?

Criterio¿Lo cumple el grafo?
Las relaciones son el dato central✅ DEPENDE_DE, ALOJA, TRABAJA_EN, REPORTÓ…
Necesito caminar profundidad > 1✅ Servidor → Servicio → Proyecto → Estudiante → Ticket
La pregunta es de impacto y propagación✅ Recorrido inverso de dependencias
En SQL necesitaría CTEs recursivas complejas✅ Cypher lo hace en 3 líneas
Costo aceptado: Si necesitáramos reportes con SUM/AVG sobre millones de registros, el grafo no es la mejor herramienta. Para ese caso, una parte del sistema debería usar SQL. En este proyecto ese límite es aceptable porque el caso de uso principal es caminar dependencias, no agregar estadísticas masivas.

services:
  neo4j:
    image: neo4j:5-community
    container_name: proyecto_neo4j

      NEO4J_apoc_import_file_enabled: "true"

    healthcheck:
      test: ["CMD", "neo4j", "status"]
      interval: 15s
      timeout: 10s

Capítulo 4: Conectar Neo4j Desktop con el contenedor Docker

¿Por qué conectar Neo4j Desktop? Neo4j Desktop (también llamado Cypher App o Neo4j Workspace) tiene un editor Cypher con autocompletado, visualización del grafo interactiva y herramientas de análisis. Es la forma más cómoda de ejecutar y demostrar las consultas del proyecto.

4.1 Descargar Neo4j Desktop

  1. Ve a https://neo4j.com/download/
  2. Descarga Neo4j Desktop para tu sistema operativo (Windows / Mac / Linux)
  3. Instala y abre la app. Te pedirá registro gratuito o saltar ese paso.

4.2 Agregar conexión remota al contenedor Docker

Importante: NO crees una base de datos nueva en Neo4j Desktop. Lo que harás es agregar una conexión remota que apunte al contenedor Docker ya corriendo.

Sigue estos pasos en Neo4j Desktop:

  1. En la pantalla principal, haz clic en "Add" (o el ícono +)
  2. Selecciona "Remote connection" (Conexión remota)
  3. Rellena los campos:
    • Name: Docker Neo4j Proyecto
    • Connect URL: neo4j://localhost:7687
    • Username: neo4j
    • Password: neo4j123
  4. Haz clic en "Connect"
  5. Si aparece un aviso de certificado SSL, acepta la conexión sin cifrar ("Use without encryption")
-- Diagrama del flujo de conexión --

 [Neo4j Desktop]         [Docker]
       |                    |
       |  Bolt (TCP 7687)   |
       |<------------------>|
  localhost:7687     container:7687
       |                    |
       |                 [Neo4j 5]
       |                 datos persistidos
       |                 en neo4j_data volume

4.3 Abrir la Cypher App en Neo4j Desktop

  1. Una vez conectado, verás la base de datos con estado "Connected" (punto verde)
  2. Haz clic en "Open" para abrir Neo4j Workspace
  3. En el panel que se abre, selecciona la pestaña "Query" (el editor Cypher)
  4. ¡Listo! Puedes escribir consultas Cypher y ver los resultados como tabla o grafo visual

4.4 Verificar que la conexión funciona

Copia y ejecuta esta consulta en Neo4j Desktop:

// Verificar conexión y versión
CALL dbms.components()
YIELD name, versions, edition
RETURN name, versions, edition;

// Debe devolver algo como:
// name: "Neo4j Kernel"  versions: ["5.x.x"]  edition: "community"

No conecta en DesktopContenedor no corriendodocker compose up -d
Error de autenticaciónContraseña incorrectaVerificar neo4j123 o resetear
Timeout de conexiónFirewall bloqueando 7687Agregar excepción en firewall
"Unable to connect"SSL requeridoUsar bolt:// en vez de neo4j://
Desktop pide TLSModo seguro activoSeleccionar "Use without encryption"

4.6 Resetear contraseña si es necesario

# Entrar al contenedor y resetear password
docker exec -it proyecto_neo4j cypher-shell -u neo4j -p neo4j123

# Una vez dentro, cambiar contraseña (si es necesario):
ALTER CURRENT USER SET PASSWORD FROM 'neo4j' TO 'neo4j123';

Operación 3 — Lectura: tickets activos y sus conexiones ③ Lectura B

¿Qué hace? Muestra los tickets no cerrados con el estudiante que los reportó, el servicio afectado y el técnico asignado (si lo hay).

// OP-3: Tickets abiertos — estudiante, servicio y técnico
MATCH (e:Estudiante)-[:REPORTÓ]->(tk:Ticket)-[:AFECTA]->(svc:Servicio)
WHERE tk.estado <> 'cerrado'
OPTIONAL MATCH (tk)-[:ASIGNADO_A]->(tec:Tecnico)
RETURN tk.id         AS ticket,
       tk.estado      AS estado,
       tk.prioridad   AS prioridad,
       e.nombre       AS reportado_por,
       svc.nombre     AS servicio_afectado,
       coalesce(tec.nombre, 'SIN ASIGNAR') AS tecnico
ORDER BY
  CASE tk.prioridad
    WHEN 'critica' THEN 1
    WHEN 'alta'    THEN 2
    WHEN 'media'   THEN 3
    ELSE 4
  END;

Operación 4 — Actualización: cerrar un ticket y actualizar estado ④ Actualización / Regla

¿Qué hace? Simula el cierre de un ticket: actualiza su estado, registra el técnico resolutor y la fecha, y luego verifica el cambio.

// OP-4a: Estado ANTES del cierre
MATCH (tk:Ticket {id:'TK-106'})
RETURN tk.id, tk.estado, tk.prioridad;
// Resultado: TK-106 | abierto | media

// OP-4b: Cerrar TK-106 y asignar técnico resolutor
MATCH (tk:Ticket {id:'TK-106'})
MATCH (tec:Tecnico {nombre:'Joshua Reinoso'})
SET tk.estado = 'cerrado',
    tk.fecha_cierre = '2026-07-08'
CREATE (tec)-[:RESUELVE {fecha_resolucion:'2026-07-08'}]->(tk)
RETURN tk.id, tk.estado, tk.fecha_cierre, tec.nombre AS resuelto_por;

// OP-4c: Verificar DESPUÉS

// OP-4d: Intentar cerrar un ticket ya cerrado (error controlado)

MATCH (tk:Ticket {id:'TK-101', estado:'cerrado'})

Operación 5 — CENTRAL: análisis de impacto en cascada ⑤ Operación central del grafo

¿Qué hace? Dado que el servidor H200 está caído, recorre el grafo para encontrar: qué servicios deja sin funcionamiento, qué proyectos quedan bloqueados, y qué estudiantes con tickets activos se ven afectados.

Esta es la consulta que justifica por qué se eligió un grafo. En SQL requeriría CTEs recursivas de 30+ líneas. En Cypher son 15 líneas elegantes.

// OP-5: ANÁLISIS DE IMPACTO EN CASCADA
// Pregunta: "Si cae el servidor H200, ¿qué pasa en la infraestructura?"

// Paso A: ¿Qué servicios aloja H200?
MATCH (srv:Servidor {nombre:'H200'})-[:ALOJA]->(svc:Servicio)
RETURN srv.nombre AS servidor, svc.nombre AS servicio_caido;
// H200 | JupyterHub
// H200 | Ollama
// H200 | Docker

// Paso B: ¿Qué otros servicios dependen de los caídos? (profundidad 1-3)
MATCH (srv:Servidor {nombre:'H200'})-[:ALOJA]->(caido:Servicio)
MATCH path = (afectado:Servicio)-[:DEPENDE_DE*1..3]->(caido)
RETURN DISTINCT
  caido.nombre    AS servicio_caido,
  afectado.nombre AS servicio_afectado_en_cascada,
  length(path)    AS saltos
ORDER BY caido.nombre, saltos;

// Paso C: ¿Qué proyectos usan esos servicios caídos?
MATCH (srv:Servidor {nombre:'H200'})-[:ALOJA]->(svc:Servicio)
MATCH (p:Proyecto)-[:USA]->(svc)
RETURN DISTINCT
  p.nombre   AS proyecto_bloqueado,
  svc.nombre AS servicio_caido,
  p.tipo     AS tipo_proyecto;

// Paso D: ¿Qué estudiantes trabajan en esos proyectos y tienen tickets activos?
MATCH (srv:Servidor {nombre:'H200'})-[:ALOJA]->(svc:Servicio)
MATCH (p:Proyecto)-[:USA]->(svc)
MATCH (e:Estudiante)-[:TRABAJA_EN]->(p)
OPTIONAL MATCH (e)-[:REPORTÓ]->(tk:Ticket)
WHERE tk.estado <> 'cerrado'
RETURN DISTINCT
  e.nombre   AS estudiante_impactado,
  p.nombre   AS proyecto,
  svc.nombre AS via_servicio,
  coalesce(tk.id, 'Sin ticket activo') AS ticket_activo
ORDER BY e.nombre;

// Paso E (MEGA-QUERY): Todo en una sola consulta de 5 saltos
MATCH path =
  (srv:Servidor {nombre:'H200'})-[:ALOJA]->(svc:Servicio)
  <-[:USA]-(p:Proyecto)
  <-[:TRABAJA_EN]-(e:Estudiante)
  -[:REPORTÓ]->(tk:Ticket)
WHERE tk.estado <> 'cerrado'
RETURN
  srv.nombre AS servidor,
  svc.nombre AS servicio,
  p.nombre   AS proyecto,
  e.nombre   AS estudiante,
  tk.id      AS ticket,
  tk.prioridad AS prioridad,
  length(path) AS saltos_en_el_grafo
ORDER BY tk.prioridad;

Operación 6 — Límite del modelo: consulta incómoda ⑥ Costo aceptado

¿Qué hace? Demuestra el límite del grafo: calcular el tiempo promedio de resolución de tickets por técnico es posible pero incómodo comparado con SQL.

// OP-6: Estadísticas de resolución por técnico
// Esto es POSIBLE en Cypher pero más natural en SQL con AVG()

MATCH (tec:Tecnico)-[r:RESUELVE]->(tk:Ticket)
RETURN
  tec.nombre  AS tecnico,
  tec.nivel   AS nivel,
  count(tk)   AS tickets_resueltos
ORDER BY tickets_resueltos DESC;

// Técnicos con mayor carga de tickets activos asignados
MATCH (tec:Tecnico)<-[:ASIGNADO_A]-(tk:Ticket)
WHERE tk.estado <> 'cerrado'
RETURN
  tec.nombre AS tecnico,
  count(tk)  AS carga_activa
ORDER BY carga_activa DESC;

// COSTO ACEPTADO: Para calcular tiempo promedio de resolución
// necesitaríamos fechas como timestamps y hacer aritmética.
// El grafo NO tiene funciones de date arithmetic tan convenientes como SQL.
// En SQL esto sería: AVG(fecha_cierre - fecha_creacion)
// Conclusión: para reportes estadísticos masivos, SQL es mejor herramienta.

  1. "Mi problema es..." — La infraestructura de IA universitaria tiene servidores, servicios y estudiantes. Cuando algo falla, el equipo técnico necesita saber el impacto en cascada.
  2. "La pregunta principal es..." — Si falla un servidor, ¿qué servicios caen, qué proyectos se bloquean y a qué estudiantes afecta con tickets activos?
  3. "Escogí esta ruta porque..." — La respuesta requiere caminar relaciones a profundidad variable. Eso es exactamente lo que un grafo hace nativamente y en SQL requeriría CTEs recursivas de 30 líneas.
  4. "Mis datos están organizados así..." — [Mostrar el grafo en Neo4j Desktop] 7 tipos de nodo, 10 tipos de relación. Las relaciones tienen propiedades como fecha, motivo y fecha_asignacion.
  5. "Estas operaciones prueban que funciona..." — [Ejecutar OP-5 en vivo] Este camino de 5 saltos: Servidor → Servicio → Proyecto → Estudiante → Ticket, en 15 líneas de Cypher.
  6. "El costo o límite principal es..." — Cálculos estadísticos masivos (SUM, AVG) son más cómodos en SQL. Si el sistema creciera y necesitara reportes de métricas, esa parte debería usar SQL o un data warehouse.
  7. "Si el sistema creciera..." — Agregaría un nodo :Red para modelar subredes, y relaciones :CONECTADO_A entre servidores para análisis de topología de red.

8.2 Preguntas frecuentes de defensa (Ruta 4)

PreguntaRespuesta sugerida
¿Por qué DEPENDE_DE va del dependiente al base? Porque la dirección modela la pregunta "este servicio necesita a este otro". Así puedo buscar hacia adelante "¿de qué depende X?" o hacia atrás "¿quién depende de X?", sin cambiar la consulta, solo invirtiendo la dirección con <-[:DEPENDE_DE]-.
¿Qué propiedad pusiste en una relación? La relación REPORTÓ tiene fecha, RESUELVE tiene fecha_resolucion, y ESCALADO_A tiene motivo. Estas propiedades viven en la relación porque describen el evento, no la entidad.
¿Qué haría difícil una consulta SQL aquí? La OP-5 requiere seguir un camino variable (1-5 saltos) de dependencias. En SQL necesitaría WITH RECURSIVE con múltiples JOINs. En Cypher es [:DEPENDE_DE*1..5].
¿Qué costo aceptaste? El grafo no es eficiente para agregaciones masivas (AVG de tiempos, sumas de métricas). Para eso, SQL es la herramienta correcta.
¿Podrías haber usado MongoDB? Podría guardar los tickets como documentos, pero las dependencias entre servicios son el dato central — eso requiere traversals que MongoDB no hace nativamente.

Capítulo 9: README del Proyecto

# Proyecto Final — Neo4j · Ruta 4 Grafo
## Sistema de Gestión de Incidencias — Infraestructura IA Universitaria

**Todos los datos son ficticios.**

## Requisitos
- Docker Desktop instalado y corriendo
- (Opcional) Neo4j Desktop para visualización

## Pasos de ejecución (desde cero)

```bash
# 1. Clonar o descomprimir el proyecto
cd proyecto-neo4j

# 2. Levantar Neo4j
docker compose up -d

# 3. Esperar ~30 segundos y verificar
docker compose ps     # Status: running
docker exec -it proyecto_neo4j neo4j status

# 4. Cargar los datos
docker exec -it proyecto_neo4j cypher-shell \
  -u neo4j -p neo4j123 \
  --file /var/lib/neo4j/import/init.cypher

# 5. Abrir Neo4j Browser
# → http://localhost:7474
# → Usuario: neo4j | Contraseña: neo4j123

# 6. (Opcional) Conectar Neo4j Desktop
# → Remote connection → bolt://localhost:7687
```

## Operaciones
Ver archivo `neo4j/consultas.cypher`. Ejecutar en orden OP-1 a OP-6.

## Salida esperada
- OP-1: 57 nodos (15 Tickets, 10 Estudiantes, 14 Servicios, 7 Servidores...)
- OP-5 (central): camino de 5 saltos Servidor→Servicio→Proyecto→Estudiante→Ticket
